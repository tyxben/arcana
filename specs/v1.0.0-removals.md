# v1.0.0 Removals — Tracking Doc

**Status**: Planning (no v1.0.0 milestone scheduled yet)
**Last updated**: 2026-04-20

This document tracks APIs and modules scheduled for **physical removal** in v1.0.0 — no compatibility shims, no silent aliases, no re-exports. Each entry records what goes, why, and the migration path users need before v1.0.0 ships.

The removal-without-shim policy is the default for v1.0.0. Exceptions require a written argument in this doc.

---

## Why remove, not shim

Every compatibility shim is a surface the constitution has to defend forever. For APIs that were deprecated because they *violated* the constitution (not because they had a better successor), keeping a shim means keeping the violation reachable. v1.0.0 is the one moment where we can cleanly remove that surface.

For APIs where both the old and new forms are constitutional and the change is purely ergonomic, a shim may be justified. For APIs where the old form encoded a prescribed strategy or topology that Principle 4 / Principle 8 rejects, removal without shim is the default.

---

## Scheduled removals

### `Runtime.team()` and the `TeamSession`/`TeamMode` machinery

**Removal policy**: physical — no compatibility shim, no alias from `team` to `collaborate`.

**Deprecated in**: v0.8.0 (2026-04-19). `DeprecationWarning` active; `CHANGELOG.md` entry at v0.8.0 announces v1.0.0 removal.

**What removal covers**:

- `Runtime.team(...)` method on `Runtime`
- `TeamSession` class
- `TeamMode` enum / `mode="shared"` / `mode="session"` parameters
- Any internal round-counting, fixed turn-order scheduler embedded in the above
- Re-exports of any of the above from `arcana/__init__.py` or the public SDK

**Why physical removal (not a shim)**:

`team(mode="shared")` bakes a framework-prescribed collaboration topology — internal round counter, fixed turn sequence, framework-owned context baton — into the runtime. Under Principle 4 (Strategy Leaps), Principle 6 (OS not Form Engine), and Principle 8 (amended per `constitution-amendment-2-collaboration-means.md`), that shape is not something the framework may prescribe. A shim from `team()` to `collaborate()` would either (a) keep the violating rounds/turn-order logic alive behind a thin wrapper, or (b) silently change behavior for anyone still calling `team()`. Both are worse than a clean removal with a migration guide.

`team(mode="session")` is already effectively subsumed by `runtime.collaborate()` + per-agent `add(...)`, so there is nothing to preserve that the new API does not already cover.

**Migration path for users**:

Full recipe lives at the bottom of `docs/guide/multi-agent.md`. One-line summary: replace `runtime.team(mode=...)` with `runtime.collaborate()` + `pool.add(name, system=..., ...)` per agent; the user's code, not the framework, drives turn order via whatever loop the user writes over `pool`.

**Pre-removal checklist** (before v1.0.0 release):

- [ ] Verify `docs/guide/multi-agent.md` migration recipe is still current against the final v0.x `collaborate()` API.
- [ ] Search `examples/` for any `runtime.team(` callsites; port to `collaborate()`.
- [ ] Search `tests/` for `runtime.team(` — expect zero tests of the deprecated API at v1.0.0.
- [ ] Confirm `DeprecationWarning` has been live for at least one PyPI release before removal (✅ v0.8.0).
- [ ] Confirm no internal callers remain (grep `src/arcana/**` for `team(` and `TeamSession`).
- [ ] Add a `BREAKING CHANGES` section to the v1.0.0 CHANGELOG entry pointing at the migration recipe.

## Not scheduled for removal (explicit non-decisions)

Listing these to pre-empt future "was this supposed to go?" questions.

- **`runtime.chain()`, `runtime.chat()`, `runtime.run()`, `runtime.run_batch()`** — all constitutional, all first-class, all stay.
- **V1 `Agent` + `AdaptivePolicy`** — separate question, not part of this removal doc. If it is ever to be removed, that needs its own entry.
- **Legacy provider factories** — stay unless a specific provider is retired upstream.

## Adding new entries

Any future "remove in v1.0.0" decision must add an entry here with:

- Removal policy (physical / aliased / shimmed) and the argument for the choice.
- Which version announced the deprecation.
- What exactly is covered.
- Why removal — tied to a constitutional principle where possible.
- Migration path, and where the full recipe lives.
- Pre-removal checklist.
