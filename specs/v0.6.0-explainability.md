# Arcana v0.6.0 "The Explainability Release" — Technical Specification

**Base**: v0.5.0 (Resilience Release)
**Theme**: 让 context management 的每个决策都变成结构化、可回放的证据
**Status**: ✅ DONE — 1337 tests, +13 new
**Constitutional check**: 每项改动通过 P2/P5/P6 + Prohibition 1/2/3 审查

---

## Overview

Arcana 定位 `runtime-first, context-governed` 的核心招牌是 context management。v0.3.0 提供了 `ContextReport`（token 开销透明），v0.5.0 提供了 Phase 0 pruning（更聪明的压缩）。此版本把招牌做成完整闭环：**每次 LLM 调用前，Arcana 能用结构化数据回答"为什么这样组 prompt"**，并支持离线回放。

两个可组合的改进：

| # | 改进 | 宪法依据 |
|---|------|----------|
| 1 | Per-message evidence in `ContextDecision` | P2 Context as Working Set / P5 Actionable Feedback |
| 2 | Prompt snapshots + replay API | P6 Runtime as OS |

**红线（明确不做）：**
- 不把 `MessageDecision` 或 relevance_score 塞进发给 LLM 的 prompt
- 不根据 decision 触发任何自动 recovery / retry / tool expansion
- 不改变现有 strategy 选择、compression 阈值、fidelity 逻辑（纯透明改动）
- Prompt snapshot 默认关（反 context / trace hoarding）

## 1. Per-message evidence — `MessageDecision`

### 问题

`ContextDecision` 之前只有 `explanation: str`（自由文本）和 `compressed_messages: list[str]`（"role:truncated_content"）。消费者无法回答：
- 哪条原始 message 被 drop / compress / 保留？
- compress 到哪个 fidelity 等级？
- 每条的 relevance score 是多少？
- 被压缩前后 token 数各是多少？
- 压缩的 reason 是什么？

### 设计

`contracts/context.py` 新增：

```python
class MessageDecision(BaseModel):
    index: int                      # 原始 messages 列表中的位置
    role: str                       # user / assistant / tool / system
    outcome: Literal["kept", "compressed", "dropped", "summarized"]
    fidelity: Literal["L0", "L1", "L2", "L3"] | None = None
    relevance_score: float | None = None
    token_count_before: int
    token_count_after: int           # 0 when dropped
    reason: str                      # 机器可读的原因字符串
```

`ContextDecision` 扩展：
- 新增 `decisions: list[MessageDecision]` — 权威 per-message 证据
- 新增 `strategy: str` — 和 `ContextReport.strategy_used` 语义对齐
- `compressed_messages` 保留（deprecated），避免破坏现有消费者
- `explanation: str` 保留作为人类可读摘要（从 decisions 派生）

### Builder 改造

`context/builder.py` 新增 `_make_decisions` helper：给定 input messages + `per_index` dict + `phase0_info` dict，输出 `list[MessageDecision]`，保证 `len(decisions) == len(messages_seen_by_strategy)`。

覆盖全部 5 种路径：

| 路径 | per-index reason |
|------|------------------|
| passthrough | `passthrough` |
| tail_preserve head/tail | `tail_preserve_head` / `tail_preserve_tail` |
| tail_preserve middle (compressed) | `tail_preserve_middle_compressed` + fidelity L0/L1/L2/L3 + score |
| tail_preserve middle (dropped by budget) | `tail_preserve_middle_dropped` |
| tail_preserve (no summary budget) | `tail_preserve_no_budget_for_middle` |
| aggressive_truncate kept | `aggressive_truncate_kept` |
| aggressive_truncate dropped | `aggressive_truncate_drop` |
| LLM summarization (middle folded) | `llm_summarized_into_single` (outcome=summarized) |

**Phase 0 集成**：`_prune_stale_tool_results` 签名从 `(msgs, count, saved)` 扩展为 `(msgs, count, saved, pruning_info)`，其中 `pruning_info: dict[int, int]` 映射 `index → original_token_count`。Reason 合并为 `stale_tool_result` 或 `stale_tool_result+<strategy_reason>`。

### Trace emission

`ConversationAgent` 抽出 `_emit_context_decision_event(state)` 辅助方法。旧的"手挑字段 metadata"改为：

```python
metadata = {
    "turn": decision.turn,
    "context_decision": decision.model_dump(),   # 全量
    "context_report": report.model_dump(),       # 全量
}
```

保持 `EventType.CONTEXT_DECISION` 不变，消费者可通过 `ContextDecision.model_validate(event.metadata["context_decision"])` 无损恢复。

## 2. Prompt snapshots + replay API

### 问题

Trace 只存 `llm_request_digest`（SHA-256 16字符）—— hash 只能判断"prompt 是否相同"，无法回答"prompt 到底是什么"。离线复现一次 LLM 调用需要的 messages/tools/model 全部缺失。

### 设计

`contracts/trace.py` 新增：

```python
class PromptSnapshot(BaseModel):
    turn: int
    model: str
    messages: list[dict[str, Any]]   # Message.model_dump()
    tools: list[dict[str, Any]]      # provider-ready schemas
    response_format: dict[str, Any] | None = None
    budget_snapshot: BudgetSnapshot | None = None

class EventType(str, Enum):
    ...
    PROMPT_SNAPSHOT = "prompt_snapshot"   # 新增
```

**不改 TraceEvent 顶层 schema** — `PromptSnapshot.model_dump()` 放入 `metadata["prompt_snapshot"]`。这样 JSONL 格式保持稳定，Rust 重写时 Serde 结构无需改。

### Gating

`RuntimeConfig.trace_include_prompt_snapshots: bool = False`（默认关）。开启后 `ConversationAgent._emit_prompt_snapshot_event` 在每个 `gateway.generate()` / `gateway.stream()` 之前发射一次 PROMPT_SNAPSHOT 事件。

关的理由：
- Prompt 内容可能含 PII / secrets
- 完整 messages 会让 trace 文件膨胀几倍
- 默认关符合 P3 "反 context hoarding" 的精神

### Replay API

`trace/reader.py` 新增：

```python
class PromptReplay(BaseModel):
    run_id: str
    turn: int
    prompt_snapshot: PromptSnapshot | None
    context_decision: ContextDecision | None
    context_report: ContextReport | None
    budget_snapshot: BudgetSnapshot | None

class TraceReader:
    def list_turns(self, run_id: str) -> list[int]: ...
    def replay_prompt(self, run_id: str, turn: int) -> PromptReplay | None: ...
```

`replay_prompt` 从 CONTEXT_DECISION + PROMPT_SNAPSHOT 事件 join 出一次 LLM 调用的完整"为什么 + 是什么"。Snapshot 关时 `prompt_snapshot=None`，但 decision 仍可用。

### CLI

`arcana trace replay <run_id> --turn N [--prompt-only|--decision-only] [--json]`

默认人类可读输出：
- Header: strategy / utilization
- per-message 表：`index  role  outcome  before→after  reason  [fidelity]  score=x.xx`
- Prompt preview：role + 前 120 字符

## 测试策略

- **Unit**: 每种 strategy 路径触发一次，断言 `len(decisions) == messages_in`
- **Outcomes**: 断言每种 outcome（kept/compressed/dropped/summarized）在至少一条路径上出现并带正确 reason / fidelity / token counts
- **Stale tool result**: 断言被 Phase 0 裁剪的 indices 在 decisions 里可见，`token_count_before > token_count_after`
- **Trace round-trip**: 写 TraceWriter → 读 TraceReader → `replay_prompt` → 断言能重建完整 ContextDecision + PromptSnapshot
- **Snapshot gating**: flag 关时 `prompt_snapshot=None`，decision 仍在

回归：1324 baseline + 13 new = 1337 passed, 0 failed。

## 宪法审查

| 审查点 | 结论 |
|--------|------|
| P2（working set） | ✅ 只记录已有决策，不改变 working set 组织 |
| P3（tools as capabilities） | ✅ 不涉及 tool 层 |
| P5（actionable feedback） | ✅ 提供诊断/调试证据，不替 LLM 决策 |
| P6（runtime as OS） | ✅ trace/observability 是 OS 核心职责 |
| Prohibition 1（不流程化） | ✅ 不改 prompt 构建顺序 |
| Prohibition 2（不做表单） | ✅ MessageDecision 是 retrospective 证据 |
| Prohibition 3（反 hoarding） | ✅ snapshot 默认关 |
| Rust 可移植 | ✅ 纯 Pydantic；JSONL 格式稳定；reader 逻辑可直接 port 到 Serde |

## 与"诊断→恢复闭环"方案 B 的对比

v0.5.0 spec 第 3 项（诊断→恢复闭环）被宪法审查否决（"框架越界指导 LLM 策略"）。本方案是同一问题域（context management 招牌深耕）的**合宪版本**：只暴露已经发生的决策数据，不向 LLM 注入任何新东西，不自动触发任何动作。用户仍是唯一的策略决定者。
