# v1.0.0 Stability — Tracking Doc

**Status**: Draft (2026-04-26)
**Owns**: the load-bearing definition of *what stable means* for v1.0.0+,
the pre-v1.0.0 work needed to deliver it, and the versioning policy that
takes effect from v1.0.0 onward.
**Supersedes / cross-references**: `specs/v1.0.0-removals.md` (breaking
removals tracker — kept as the canonical removals list; this doc adds
non-breaking work + policy).

This spec exists because external feedback (Roboot integration, 2026-04)
showed that "what is stable" is currently only knowable by reading
source. That is the constitutional violation Principle 5
(*Auditability*) is supposed to prevent for users, not just for runtime
internals. v1.0.0 is the moment to fix it.

---

## 1. Stable public surface (effective v1.0.0)

These names ship a stability promise: rename / removal / signature
breakage is a major bump, not a minor.

### 1.1 SDK top-level (`arcana.*`)

- `arcana.run(...)` — single-shot task
- `arcana.Runtime` — resource container
- `arcana.Budget` — budget configuration
- `arcana.AgentConfig` — agent configuration
- `arcana.RuntimeConfig` — runtime configuration
- `arcana.tool` — `@arcana.tool` decorator
- `arcana.BatchResult` — batch result wrapper
- `arcana.ChainStep` — pipeline step
- `arcana.ChainResult` — pipeline result
- `arcana.Message`, `arcana.MessageRole` — re-exported from
  `arcana.contracts.llm`. The top-level form is preferred for user code
  even though contracts/ is the canonical definition.

### 1.2 Runtime methods (`Runtime.*`)

- `Runtime.run(...)` / `Runtime.run_batch(...)`
- `Runtime.chat(...)` → `ChatSession`
- `Runtime.chain(...)` → `ChainResult`
- `Runtime.collaborate(...)` → `AgentPool` (replaces deprecated
  `Runtime.team()`)
- `Runtime.session(...)` — manual session control
- `Runtime.close()` — lifecycle

### 1.3 ChatSession surface

- `ChatSession.send(...)`, `ChatSession.stream(...)`
- `ChatSession.history` (read-only dict copy)
- `ChatSession.total_cost_usd`, `ChatSession.total_tokens`,
  `ChatSession.turn_count`
- `ChatSession.max_history`
- `ChatSession.seed_history(...)` *(to be added pre-v1.0.0 — see §3.1)*
- Async context manager (`async with`)

### 1.4 Contracts (`arcana.contracts.*`)

All names below are stable as data contracts: field rename = major bump,
field addition with default = minor bump.

- `arcana.contracts.tool` — `ToolSpec`, `ToolCall`, `ToolResult`,
  `ToolError`, `ToolErrorCategory`, `SideEffect`, `ASK_USER_TOOL_NAME`
- `arcana.contracts.llm` — `Message`, `MessageRole`, `LLMRequest`,
  `LLMResponse`, `ContentBlock`, `ModelConfig`
- `arcana.contracts.turn` — `TurnFacts`, `TurnAssessment`
- `arcana.contracts.context` — `ContextBlock`, `ContextBudget`,
  `ContextDecision`, `MessageDecision`
- `arcana.contracts.diagnosis` — `DiagnosticBrief`, `ErrorCategory`
- `arcana.contracts.streaming` — `StreamEvent`, `StreamEventType`
  *(with §4.2 cleanup)*
- `arcana.contracts.runtime` — `RuntimeConfig`
- `arcana.contracts.cognitive` — recall/pin/unpin contracts

### 1.5 Tool authoring

- `@arcana.tool` decorator surface
- `ToolSpec` field set
- `SideEffect` enum values
- `ToolErrorCategory` enum values + retry-eligibility rule

### 1.6 CLI

- `arcana run`, `arcana init`, `arcana trace {show, replay, explain,
  flow, pool-replay}` — invocation form, exit codes, primary flag set.
  Output formatting is best-effort, not stability-promised.

## 2. Internal — NOT stable

These imports work today, but break without notice across releases. User
code that reaches into them assumes the maintenance burden.

- `arcana.runtime.*` — `ConversationAgent`, `Agent`, internal handlers
- `arcana.gateway.*`, `arcana.tool_gateway.*` — provider/tool plumbing
- `arcana.context.*` — `WorkingSetBuilder` and compression internals
- `arcana.routing.*`, `arcana.eval.*`, `arcana.trace.*`,
  `arcana.streaming.*`, `arcana.multi_agent.*` — implementation modules
- Anything starting with `_` — period.

The asymmetric cases:

- `arcana.contracts.*` — **stable** even though it's a sub-package
  (contracts are the data layer's stable surface)
- `arcana.runtime.conversation.Message` — **NOT a stable import path**.
  `Message` is reachable there as an internal import, not a re-export.
  Users importing it from this path will break when the internal
  `from arcana.contracts.llm import Message` line moves. Use
  `arcana.Message` or `arcana.contracts.llm.Message`. Pre-v1.0.0
  cleanup at §3.2 makes this explicit.

## 3. Pre-v1.0.0 work (non-breaking, ships in v0.9.x)

Implementation order is roughly ROI-descending. Each item is independent
and parallelizable.

### 3.1 `ChatSession.seed_history(messages)` (#1 from feedback)

**Problem**: cold-start chatbots need to inject prior conversation
history. Current `ChatSession.history` returns a dict copy and there is
no public mutator, so users are forced to poke `session._messages`. That
*is* a constitutional smell — Principle 5 wants every state mutation
auditable, and reaching past private API bypasses the trace.

**Surface**:
```python
def seed_history(
    self,
    messages: list[Message] | list[dict[str, str]],
) -> None: ...
```

- Accepts `Message` (canonical) or dict-form `{"role": ..., "content":
  ...}` (convenience).
- Idempotent only by user discipline — calling twice extends.
- Emits a new `EventType.HISTORY_SEEDED` trace event with
  `{message_count, digests}` so the seed is auditable.
- Available before *or* after `send()` — but documented that mid-stream
  seeding may collide with active compression state. No framework guard
  rail; the user owns the timing call.

**Why public, not optional**: cold-start history is a standard pattern.
Forcing private-attribute access is the controllability theater
Prohibition 2 forbids — the framework currently *says* "use my public
API" but the public API doesn't cover the case.

### 3.2 `Message` canonical path (#2)

**Problem**: `from arcana.runtime.conversation import Message` works
because of an internal `from ... import` line at module top. Users who
discover that path are coupling to an implementation detail.

**Fix**:
- `arcana/runtime/conversation.py` module docstring: explicit note that
  `Message` and `MessageRole` are imported here for internal use; user
  code should import them from `arcana.contracts.llm` (canonical) or
  the top-level `arcana` namespace (preferred).
- Add `__all__ = [...]` to `runtime/conversation.py` *excluding*
  `Message` and `MessageRole` so `from arcana.runtime.conversation
  import *` does not advertise them.
- Confirm `arcana/__init__.py` re-exports `Message` / `MessageRole`
  from `arcana.contracts.llm` at the top level. If yes, document. If
  no, add — the top-level form is the *preferred* user import.

**Not done**: physical removal. The internal import is real and used; we
just stop endorsing the path as a public API.

### 3.3 Stability doc — user-facing copy of §1–2 (#3)

**Problem**: §1 and §2 of this spec are the reference. Users need a
shorter, more readable form in `docs/guide/`.

**Fix**: `docs/guide/stability.md` — distilled version of §1–2 plus
"how to know if your import is stable" (rule of thumb: if it's in the
list, yes; if it has an underscore, no; if it's `arcana.runtime.*` or
similar internal module, no).

**Why high ROI**: pure documentation, no code risk. Resolves the
load-bearing complaint underneath items #1, #2, #4 simultaneously —
users had to read source because the doc didn't exist.

### 3.4 PR template — migration note requirement (#5)

**Problem**: breaking renames historically shipped without migration
recipes in CHANGELOG. v0.9.0 fixed this for `ToolErrorCategory`; we want
the pattern enforced.

**Fix**: add to `.github/pull_request_template.md`:

> **If this PR breaks a public surface (per `specs/v1.0.0-stability.md`),
> the CHANGELOG entry must include a `Migration` section with
> before/after code snippets.**

Optional follow-up: a CI check that grep-fails if a CHANGELOG entry
labelled `BREAKING` lacks a `Migration` heading.

### 3.5 Provider quirks doc (#6)

**Problem**: providers have known quirks (GLM-4-flash tool-calling
needs more explicit prompts; MiniMax auto-degrades on certain tool
schemas; etc.). Currently this knowledge lives in user CLAUDE.md files
and oral tradition.

**Constitutional non-fix**: the feedback proposed Arcana auto-rewrite
user system prompts per provider. **This is rejected** — Principle 4
(Strategy Leaps) and Principle 6 (OS not Form Engine) both forbid the
framework deciding *how* to prompt. The framework decides *whether* a
capability is supported (already done via `ProviderProfile`), not how
the user phrases their goals.

**Fix**: enhance `docs/guide/providers.md` with a per-provider "known
quirks + recommended prompt patterns" section. This puts the knowledge
where users can find it, while leaving the prompt itself in user
control.

## 4. v1.0.0 breaking work

### 4.1 Existing — `Runtime.team()` removal

Tracked in `specs/v1.0.0-removals.md`. Status: deprecated since v0.8.0,
warning live, pre-removal checklist in the linked doc. No change here.

### 4.2 New — `StreamEventType` consolidation (#4 from feedback)

**Problem**: `StreamEventType.LLM_CHUNK` (line 22 of
`contracts/streaming.py`, Runtime/Graph layer) and
`StreamEventType.TEXT_DELTA` (line 33, "structured agent events" v0.3.0)
**both exist** as separate enum members covering largely the same
concept. This is two streaming protocols co-existing, not just a naming
issue. Adding an alias makes it worse.

**Decision (proposed)**: keep `TEXT_DELTA` (more semantically obvious +
matches the structured event family `THINKING` / `TOOL_START` /
`TURN_END`), remove `LLM_CHUNK` and `LLM_COMPLETE` and `LLM_THINKING`.
Migrate `LLM_EVENTS` filter set + `STREAM_MODE_FILTERS["llm"]` to the
structured family.

**Risk**: this is a breaking change visible to anyone listening on
`LLM_CHUNK`. CHANGELOG `Migration` section required.

**Owner**: needs a separate spec — `specs/streaming-consolidation.md` —
before code lands. The decision above is the *proposal*, not yet
ratified.

## 5. Versioning policy (effective from v1.0.0)

Pre-v1.0.0, Arcana follows a relaxed semver where minor bumps may
include breaking changes if accompanied by a `DeprecationWarning` for
at least one prior release. From v1.0.0 onward:

- **Major (X.0.0)**: any break to the surface listed in §1. Field
  rename, signature change, removal, behavior change visible to user
  code.
- **Minor (0.X.0)**: additive only. New public method, new contract
  field with default, new provider, new event type.
- **Patch (0.0.X)**: bug fix, internal refactor invisible to public
  surface.

Internal modules (§2) get no version promise.

---

## 6. Deprecation policy

For changes that *will* be made to the stable surface in a future major
bump:

- Minimum **one minor release** with `DeprecationWarning` before
  removal (was the rule de-facto since v0.7.0; now codified).
- The CHANGELOG entry that adds the deprecation must include a
  `Migration` recipe.
- `specs/v1.0.0-removals.md` (or successor `vX.0.0-removals.md`) tracks
  the next-major removal list.
- The successor name and its replacement path must be live in the same
  release where deprecation lands — no "use the future API once we
  build it."
