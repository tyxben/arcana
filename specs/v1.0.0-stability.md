# v1.0.0 Stability ‚Äî Tracking Doc

**Status**: Draft (2026-04-26)
**Owns**: the load-bearing definition of *what stable means* for v1.0.0+,
the pre-v1.0.0 work needed to deliver it, and the versioning policy that
takes effect from v1.0.0 onward.
**Supersedes / cross-references**: `specs/v1.0.0-removals.md` (breaking
removals tracker ‚Äî kept as the canonical removals list; this doc adds
non-breaking work + policy).

This spec exists because external feedback (Roboot integration, 2026-04)
showed that "what is stable" is currently only knowable by reading
source. That is the constitutional violation Principle 5
(*Auditability*) is supposed to prevent for users, not just for runtime
internals. v1.0.0 is the moment to fix it.

---

breakage is a major bump, not a minor.

- `arcana.ChainResult` ‚Äî pipeline result

### 1.5 Tool authoring

- `@arcana.tool` decorator surface
- `ToolSpec` field set
- `SideEffect` enum values
- `ToolErrorCategory` enum values + retry-eligibility rule

**Additive history** (sanctioned minor changes under ¬ß1.4 "field addition
with default = minor bump"):

- `ToolSpec.provenance: ToolProvenance | None = None` ‚Äî capability origin /
  server identity, consumed by the permission policy (`origins` /
  `server_names` dimensions) and surfaced to trace (kimi-code slice,
  2026-05-30). The new `ToolProvenance` type is exported from
  `arcana.contracts` but, like the `arcana.contracts.permission` surface it
  serves, is intentionally **not yet** enumerated in ¬ß1.4 ‚Äî so the provenance
  frontier stays free to evolve (authority / auth-context / approval-policy
  land with the later ProtocolBridge work).
- `MCPToolSpec.annotations: dict[str, Any] | None = None` ‚Äî raw passthrough of
  the MCP `annotations` object (`readOnlyHint` / `destructiveHint` / ...),
  consumed by the imported-capability exposure gate to decide conservative
  side-effect classification (kimi-code exposure-gate slice, 2026-06-02).
  `MCPToolSpec` lives in `arcana.contracts.mcp`, which ‚Äî like the permission /
  provenance surfaces it sits beside ‚Äî is **not yet** enumerated in ¬ß1.4, so the
  MCP contract stays free to evolve.
- `EventType.CAPABILITY_ADMISSION` ‚Äî new trace event recording an imported
  capability's admission decision (admitted / downgraded / refused) as labeled
  v3.5 evidence. Enum **addition** to `arcana.contracts.trace.EventType`;
  additive, consumers that switch on `EventType` must tolerate unknown members.
  The event metadata now includes `capability_digest` so dynamic MCP discovery
  can prove each refreshed capability was re-hashed before exposure.
- `EventType.PROTOCOL_DISCOVERY` ‚Äî new trace event recording adapter-level
  protocol discovery decisions (MCP initial discovery, dynamic refresh,
  ignored notifications, and failures). Enum **addition**; the event is
  separate from `CAPABILITY_ADMISSION` so discovering a protocol source never
  implies trust or exposure approval. Metadata is shaped by
  `ProtocolDiscoveryRecord`.
- `EventType.GUARDRAIL_DECISION` and `GuardrailDecisionRecord` ‚Äî new trace
  contract for guardrail boundary evidence (block / redact / warn / approval
  requirement). Tool-call guardrails now emit this event from the
  `ToolGateway` boundary; the record deliberately carries boundary outcome
  metadata, not replacement goals or hidden workflow state.
- `arcana.contracts.guardrail` ‚Äî new tool-call guardrail contracts:
  `GuardrailAction`, `ToolGuardrailRequest`, and `GuardrailDecision`.
  The module is exported from `arcana.contracts` but is **not** exported at
  top-level `arcana.*`; lifecycle-hook semantics remain free to harden before
  any top-level stability promise.
- `Runtime(guardrails=[...])` ‚Äî additive constructor argument for tool-call
  boundary guardrails. The configured guardrails are inherited by runtime,
  MCP, and run-scoped tool gateways.
- `arcana.contracts.skill` ‚Äî new Skills v1 contract module with `SkillSpec`,
  `SkillSelectionRecord`, and `SkillRegistry`. The module is exported from
  `arcana.contracts` but is **not** exported at top-level `arcana.*`; selection
  semantics remain free to harden before any top-level stability promise.
- `ContextDecision.skill_selections: list[SkillSelectionRecord] = []` plus
  `ContextReport.skills_loaded / skills_available / skills_tokens` ‚Äî additive
  evidence fields showing which selected skills entered the working set.
- `RuntimeConfig.skill_paths: list[str] = []` and `RuntimeConfig.max_skills: int
  = 3` ‚Äî additive defaults; no path scanning or prompt injection occurs unless
  callers opt in.
- `Runtime.run(skills: list[str] | None = None)` ‚Äî keyword-only additive
  parameter for explicit per-run skill selection. Runtime skill context still
  requires configured `skill_paths`.
- `arcana.run(skill_paths: list[str] | None = None, skills: list[str] | None =
  None)` ‚Äî keyword-only additive SDK helper parameters mirroring the Runtime
  behavior for one-shot runs.

### 1.6 CLI

- `arcana run`, `arcana init`, `arcana trace {show, replay, explain,
  flow, pool-replay}` ‚Äî invocation form, exit codes, primary flag set.
  Output formatting is best-effort, not stability-promised.

## 2. Internal ‚Äî NOT stable

These imports work today, but break without notice across releases. User
code that reaches into them assumes the maintenance burden.

- `arcana.runtime.*` ‚Äî `ConversationAgent`, `Agent`, internal handlers
- `arcana.gateway.*`, `arcana.tool_gateway.*` ‚Äî provider/tool plumbing
- `arcana.context.*` ‚Äî `WorkingSetBuilder` and compression internals
- `arcana.routing.*`, `arcana.eval.*`, `arcana.trace.*`,
  `arcana.streaming.*`, `arcana.multi_agent.*` ‚Äî implementation modules
- Anything starting with `_` ‚Äî period.

The asymmetric cases:

- `arcana.contracts.*` ‚Äî **stable** even though it's a sub-package
  (contracts are the data layer's stable surface)
- `arcana.runtime.conversation.Message` ‚Äî **NOT a stable import path**.
  `Message` is reachable there as an internal import, not a re-export.
  Users importing it from this path will break when the internal
  `from arcana.contracts.llm import Message` line moves. Use
  `arcana.Message` or `arcana.contracts.llm.Message`. Pre-v1.0.0
  cleanup at ¬ß3.2 makes this explicit.

## 3. Pre-v1.0.0 work (non-breaking, ships in v0.9.x)

Implementation order is roughly ROI-descending. Each item is independent
and parallelizable.

### 3.1 `ChatSession.seed_history(messages)` (#1 from feedback)

**Problem**: cold-start chatbots need to inject prior conversation
history. Current `ChatSession.history` returns a dict copy and there is
no public mutator, so users are forced to poke `session._messages`. That
*is* a constitutional smell ‚Äî Principle 5 wants every state mutation
auditable, and reaching past private API bypasses the trace.

**Surface**:
```python
def seed_history(
    self,
    messages: list[Message] | list[dict[str, str]],
) -> None: ...
```

- Accepts `Message` (canonical) or dict-form `{"role": ..., "content":
  ...}` (convenience).
- Idempotent only by user discipline ‚Äî calling twice extends.
- Emits a new `EventType.HISTORY_SEEDED` trace event with
  `{message_count, digests}` so the seed is auditable.
- Available before *or* after `send()` ‚Äî but documented that mid-stream
  seeding may collide with active compression state. No framework guard
  rail; the user owns the timing call.

**Why public, not optional**: cold-start history is a standard pattern.
Forcing private-attribute access is the controllability theater
Prohibition 2 forbids ‚Äî the framework currently *says* "use my public
API" but the public API doesn't cover the case.

### 3.2 `Message` canonical path (#2)

**Problem**: `from arcana.runtime.conversation import Message` works
because of an internal `from ... import` line at module top. Users who
discover that path are coupling to an implementation detail.

**Fix**:
- `arcana/runtime/conversation.py` module docstring: explicit note that
  `Message` and `MessageRole` are imported here for internal use; user
  code should import them from `arcana.contracts.llm` (canonical) or
  the top-level `arcana` namespace (preferred).
- Add `__all__ = [...]` to `runtime/conversation.py` *excluding*
  `Message` and `MessageRole` so `from arcana.runtime.conversation
  import *` does not advertise them.
- Confirm `arcana/__init__.py` re-exports `Message` / `MessageRole`
  from `arcana.contracts.llm` at the top level. If yes, document. If
  no, add ‚Äî the top-level form is the *preferred* user import.

**Not done**: physical removal. The internal import is real and used; we
just stop endorsing the path as a public API.

### 3.3 Stability doc ‚Äî user-facing copy of ¬ß1‚Äì2 (#3)

**Problem**: ¬ß1 and ¬ß2 of this spec are the reference. Users need a
shorter, more readable form in `docs/guide/`.

**Fix**: `docs/guide/stability.md` ‚Äî distilled version of ¬ß1‚Äì2 plus
"how to know if your import is stable" (rule of thumb: if it's in the
list, yes; if it has an underscore, no; if it's `arcana.runtime.*` or
similar internal module, no).

**Why high ROI**: pure documentation, no code risk. Resolves the
load-bearing complaint underneath items #1, #2, #4 simultaneously ‚Äî
users had to read source because the doc didn't exist.

### 3.4 PR template ‚Äî migration note requirement (#5)

**Problem**: breaking renames historically shipped without migration
recipes in CHANGELOG. v0.9.0 fixed this for `ToolErrorCategory`; we want
the pattern enforced.

**Fix**: add to `.github/pull_request_template.md`:

> **If this PR breaks a public surface (per `specs/v1.0.0-stability.md`),
> the CHANGELOG entry must include a `Migration` section with
> before/after code snippets.**

Optional follow-up: a CI check that grep-fails if a CHANGELOG entry
labelled `BREAKING` lacks a `Migration` heading.

### 3.5 Provider tool-calling hints ‚Äî infrastructure + docs (#6)

**Problem**: providers have known tool-calling quirks (GLM-4-flash
needs more explicit instructions; MiniMax auto-degrades on certain
tool schemas; etc.). Currently this knowledge lives in user CLAUDE.md
files and oral tradition. Every Arcana adopter re-discovers it.

**Constitutional analysis (revised 2026-04-26)**: the original
feedback proposed framework auto-rewriting of user system prompts
per provider. The first instinct was to reject as Principle 4 +
Principle 6 violation ‚Äî but that conflates the **goal** (absorb
provider tool-calling quirks) with the **mechanism**. Re-decomposing:

- ‚ùå Hardcoded "GLM auto-appends fixed prompt invisibly" ‚Äî framework
  decides *how* to talk to the LLM. Violates Principle 4.
- ‚öÝÔ∏è Framework ships per-provider default hints, opt-out ‚Äî gray zone:
  default values *are* the framework taking a position on
  prompt strategy. Cross-version default changes become silent
  behavior changes. Reject.
- ‚úÖ Framework provides an **infrastructure slot** (default empty),
  user supplies content, the slot's value is rendered as a separate
  system block when tools are bound and traced via `PromptSnapshot`.
  This is the same shape as existing capability translation
  (`ProviderProfile` schema downgrade, content block format
  conversion) ‚Äî translates user *intent* to provider-acceptable
  *form*, without the framework authoring strategy.

**Fix ‚Äî two layers**:

#### 3.5a Infrastructure slot

A user-controlled hint that is rendered as a separate system message
block when tools are bound and the value is non-empty. Default empty
‚Äî no behavior change for existing users.

Concrete surface (final spelling TBD during implementation):

```python
# Per-provider granularity (preferred ‚Äî pain is provider-specific)
runtime = Runtime(
    providers={"glm": glm_key, "openai": openai_key},
    provider_profiles={
        "glm": ProviderProfile(
            tool_calling_hint=(
                "When you need to call a tool, output ONLY the tool "
                "call JSON, no commentary."
            ),
        ),
    },
)
```

Rendering contract:
- Hint is appended as a **separate** system block, not merged into
  the user's system prompt ‚Äî the user's authored text is unchanged.
- Only rendered when `tools` is non-empty for the request.
- Visible in `PromptSnapshot`; emits no special event (it is part of
  the request, fully captured by existing snapshot machinery).
- No-op when value is empty/None.

What stays out of scope:
- **No framework-shipped defaults** for any provider. The decision
  about what GLM "should" be told is the user's, not the
  framework's. (Layer 3.5b ships *suggestions* as documentation that
  the user copy-pastes intentionally.)
- **No auto-rewriting** of the user's system prompt. The hint is an
  additional block, not a mutation of user content.

#### 3.5b Community-recommended hints (docs)

`docs/guide/providers.md` per-provider section listing observed
quirks plus *suggested* hint text users can copy-paste into
`tool_calling_hint`. These are versioned in docs, not in code ‚Äî
updating them is a docs PR, not a behavior change in a release.

This is the "where to look" answer to the original feedback's pain
point. The user finds the recommendation in one place instead of
re-discovering it.

#### Why this is constitutional after the revision

- Principle 4 (Strategy Leaps): framework provides plumbing (the
  slot), user owns the content. No framework opinion on prompt
  strategy ships in code.
- Principle 5 (Auditability): the hint's contribution to any prompt
  is captured by `PromptSnapshot` ‚Äî fully traceable.
- Principle 6 (OS not Form Engine): a typed slot for an additional
  system block is OS-shaped (infrastructure for a thing that
  happens), not Form-shaped (prescribing what the LLM should say).
- Prohibition 1 (No Premature Structuring): default-empty means no
  behavior change for anyone who doesn't opt in ‚Äî the slot is added
  in response to a real, observed need, not a speculative one.

## 4. v1.0.0 breaking work

### 4.1 Existing ‚Äî `Runtime.team()` removal

Tracked in `specs/v1.0.0-removals.md`. Status: deprecated since v0.8.0,
warning live, pre-removal checklist in the linked doc. No change here.

### 4.2 Closed ‚Äî `StreamEventType` consolidation (#4 from feedback)

**Status**: closed as no-op for v1.0.0. Original problem statement was
based on a misdiagnosis (audited 2026-04-28).

**Original framing**: `LLM_CHUNK` and `TEXT_DELTA` were assumed to be
two parallel streaming protocols whose duplication needed resolving
before v1.0.0 froze the surface.

**What the code actually shows**: there is one live streaming protocol.
- `LLM_CHUNK` / `LLM_COMPLETE` / `LLM_THINKING` are emitted by
  `runtime/conversation.py`, covered by tests, and documented in
  `docs/guide/api.md`.
- The "structured agent events" group from v0.3.0 was partially
  wired: `TOOL_START` / `TOOL_END` / `TURN_END` / `CONTEXT_REPORT`
  are emitted; `TEXT_DELTA` and `THINKING` have zero references
  outside `contracts/streaming.py` (never emitted, never tested,
  never documented).

So there are not two competing protocols ‚Äî there is one protocol plus
two inert enum members from a planning artifact. The original
proposal (remove `LLM_CHUNK` family, migrate to `TEXT_DELTA`) would
have removed the live protocol in favor of dead code.

**Decision**: leave the two inert members in place. They cost
nothing. Per ¬ß5, removing public enum members is a major break, so
removal would need its own deprecation cycle (warn in a future minor,
remove in v2.0.0). The motivation for that cycle does not exist
today. v1.0.0's only breaking item is ¬ß4.1.

If a real second protocol is ever needed (e.g. typed payload
`StreamEvent` variants), that is a separate spec ‚Äî not ¬ß4.2.

## 5. Versioning policy (effective from v1.0.0)

Pre-v1.0.0, Arcana follows a relaxed semver where minor bumps may
include breaking changes if accompanied by a `DeprecationWarning` for
at least one prior release. From v1.0.0 onward:

- **Major (X.0.0)**: any break to the surface listed in ¬ß1. Field
  rename, signature change, removal, behavior change visible to user
  code.
- **Minor (0.X.0)**: additive only. New public method, new contract
  field with default, new provider, new event type.
- **Patch (0.0.X)**: bug fix, internal refactor invisible to public
  surface.

Internal modules (¬ß2) get no version promise.

---

## 6. Deprecation policy

For changes that *will* be made to the stable surface in a future major
bump:

- Minimum **one minor release** with `DeprecationWarning` before
  removal (was the rule de-facto since v0.7.0; now codified).
- The CHANGELOG entry that adds the deprecation must include a
  `Migration` recipe.
- `specs/v1.0.0-removals.md` (or successor `vX.0.0-removals.md`) tracks
  the next-major removal list.
- The successor name and its replacement path must be live in the same
  release where deprecation lands ‚Äî no "use the future API once we
  build it."
