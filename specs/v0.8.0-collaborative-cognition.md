# Arcana v0.8.0 "The Collaborative Cognition Release" — Technical Specification

**Base**: v0.7.0 (Cognitive Primitives Release)
**Theme**: Multi-agent pools as infrastructure; each agent is an independent cognitive instance
**Status**: 📝 PROPOSED
**Depends on**: Principle 8 (Agent Autonomy in Collaboration) + Principle 9 (Cognitive Primitives as Services)

---

## Overview

v0.7.0 made cognition operable by a single LLM: `recall` and `pin` let the LLM
work on its own reasoning state. v0.8.0 extends that to **multi-agent** settings
without becoming an orchestration framework.

The working tree already contains the unreleased infrastructure: `AgentPool`,
`Channel`, `SharedContext`, and `runtime.collaborate()`. v0.8.0 formalizes that
infrastructure, wires per-agent cognitive primitives through it, and adds
pool-aware observability — while holding the constitutional line that the
framework never dictates topology, turn order, or stop conditions.

**The Arcana positioning for multi-agent becomes:**

> LangGraph / AutoGen / CrewAI offer orchestration primitives (graphs,
> group chat, role hierarchy) that decide *what happens next* in a
> multi-agent run. Arcana's constitution (Principle 8 + the four prohibitions)
> forbids that. Instead, Arcana offers **per-agent cognition**: each agent in
> a pool is an independent cognitive instance with its own pin state, its own
> recall history, and its own compression budget. User code does the
> orchestration with plain Python (`async for`, `if`, `await`); the framework
> gives agents a shared bus, shared state, and individual reasoning tools.
>
> If you want a graph, use LangGraph. If you want cognition per agent —
> recallable history, pinned conclusions, trace-addressable decisions — that
> is Arcana.

---

## Scope

v0.8.0 ships three coordinated changes:

| # | Area | What it is | Why it is load-bearing |
|---|------|------------|------------------------|
| 1 | Formalize infrastructure | `AgentPool` / `Channel` / `SharedContext` + `runtime.collaborate()` | Lets user code assemble multi-agent flows in plain Python without a DSL |
| 2 | Per-agent cognition | Each `pool.add(...)` accepts `cognitive_primitives=[...]`; each agent owns its own `PinState` / `CognitiveHandler` | The *differentiated* value vs other multi-agent frameworks |
| 3 | Pool-level observability | Trace events carry `source_agent` metadata; `arcana trace pool-replay` | Preserves v0.6.0's replay story across multi-agent runs |

It also fixes two known bugs in the working-tree infrastructure (see
"Known bugs & fix decisions").

It does **not** ship Tier 2 cognitive primitives (`branch`, `anchor`, `hint`)
— those remain on the v0.9.0 backlog.

## Design invariants (apply across all three areas)

### 1. Pool provides infrastructure only

`AgentPool` exposes named `ChatSession`s, one shared `Channel`, one shared
`SharedContext`, and one shared `BudgetTracker`. It does **not** provide:

- Turn order (user code decides)
- Topology or graph (user code decides)
- Role hierarchy (agent prompts define roles; framework does not enforce)
- Stop conditions (user code decides)
- Automatic routing of one agent's output to another agent

This matches Principle 8 verbatim.

### 2. Per-agent cognitive isolation

Cognitive state is **per session, never shared across pool members**:

- Agent A's `PinState` is not visible to Agent B
- Agent A's `recall(turn=3)` returns A's turn 3, not the pool's "turn 3"
- Cognitive primitives must be enabled per agent via `pool.add(..., cognitive_primitives=[...])`
- Pool-level default can be passed at `runtime.collaborate(cognitive_primitives=[...])` and inherited by each `add` unless overridden

Rationale: cognitive state is a private reasoning artifact. Sharing pins
between agents would be the framework making a judgment call about what one
agent's reasoning is worth to another — a constitutional violation.

### 3. Explicit communication, never implicit

Agents communicate only through:

- **`SharedContext`** — explicit key-value set/get, user code decides what to write and read
- **`Channel`** — explicit send/receive with sender/recipient names
- Normal function returns (user code passes `result.content` to the next `send()`)

There is no "conversation continuation" between pool agents. If agent B should
see agent A's last message, user code writes `shared.set("A_output", a.content)`
and agent B reads it in its next prompt.

### 4. Full trace coverage with `source_agent`

Every trace event emitted during a pool run carries `metadata["source_agent"]`
(the pool-name of the originating agent). This applies to:

- `LLM_REQUEST` / `LLM_RESPONSE`
- `TOOL_CALL` / `TOOL_RESULT`
- `COGNITIVE_PRIMITIVE` (added v0.7.0)
- `CONTEXT_DECISION` (from v0.6.0's `MessageDecision` stream)

The `TraceEvent` schema is **unchanged** — `source_agent` lives in `metadata`,
so v0.6.0/v0.7.0 trace consumers keep working unmodified.

### 5. Shared budget, per-agent cognition budget

The pool has one shared `BudgetTracker` — all agents' LLM calls debit it
together, and one agent overspending starves the rest (intentional: it's what
"shared budget" means). Per-agent cognitive budgets (`pin_budget_fraction`)
remain per session, because pins occupy that agent's own context window and
nothing else.

### 6. No changes to single-agent API

`runtime.chat()`, `runtime.run()`, `runtime.chain()`, and the v0.7.0 cognitive
primitives work identically. v0.8.0 is additive.

## 1. `AgentPool` — formalized infrastructure

### Current working-tree API (to be kept)

```python
async with runtime.collaborate(budget=Budget(max_cost_usd=5.0)) as pool:
    planner = pool.add("planner", system="You plan", provider="deepseek")
    executor = pool.add("executor", system="You execute", tools=[shell_tool])

    plan = await planner.send("Plan a migration for project X")
    result = await executor.send(f"Execute: {plan.content}")
```

- `pool.add(name, *, system, tools?, provider?, model?, max_history?, cognitive_primitives?)` — creates a named `ChatSession`; shares pool budget
- `pool.channel` — `Channel` (name-addressed async messaging)
- `pool.shared` — `SharedContext` (thread-safe key-value store)
- `pool.agents` — `dict[str, ChatSession]` (copy)
- `pool.close()` / `async with`

### Constitutional posture of each piece

| API | Safe? | Why |
|-----|-------|-----|
| `pool.add(name=...)` | ✅ | Named sessions; user picks names and prompts |
| `pool.channel` | ✅ | Dumb pipe (post/receive); user decides when to read |
| `pool.shared` | ✅ | Opaque key-value store; user decides what's in it |
| `pool.agents` | ✅ | Read-only snapshot |
| `pool.close()` / `__aexit__` | ✅ | Releases queues, clears shared state, flushes trace — no orchestration |
| Shared budget | ✅ | Infrastructure constraint, not strategy |
| Per-agent `cognitive_primitives` | ✅ | Principle 9 — opt-in services for the LLM |

### What `__aexit__` does and does not do

**Does:**
- Close all `ChatSession`s (releases provider connections)
- Drain and clear `Channel` queues
- Clear `SharedContext`
- Flush pending trace events

**Does not:**
- Collect "final answers" from any agent
- Decide whether the run succeeded
- Emit a summary event
- Auto-persist any state

This matches `ChatSession.__aexit__`.

## 2. Per-agent cognitive primitives

### API

```python
async with runtime.collaborate(cognitive_primitives=["recall", "pin"]) as pool:
    # Both agents inherit pool default
    planner = pool.add("planner", system="...")
    executor = pool.add("executor", system="...", cognitive_primitives=["pin"])
    # planner has recall + pin; executor only pin
```

### Contract

`pool.add(...)` accepts `cognitive_primitives: list[str] | None = None`:
- `None` (default) → inherit pool-level default
- `[]` → explicitly opt out for this agent
- `["recall"]` / `["pin"]` / `["recall", "pin"]` → explicit opt in

Pool-level default comes from `runtime.collaborate(cognitive_primitives=[...])`:
- `None` / omitted → default empty (no primitives enabled anywhere)

### Per-agent state isolation

Every `ChatSession` created via `pool.add` instantiates its own:
- `CognitiveHandler` (from `arcana.runtime.cognitive`)
- `PinState` (registry of active pins for *this agent*)
- Turn log (for `recall`)

No code path writes one agent's pin to another agent's `PinState`. Budget cap
(`pin_budget_fraction`) is computed against that agent's context window, not
the pool's shared budget.

### Tool schema visibility

If an agent has `cognitive_primitives=["recall"]`, the `recall` tool appears in
**that agent's** tool schema only. Other pool agents without the primitive do
not see it — keeping working-set tool schemas minimal (Principle 3).

### Tool name collisions

An agent's user-supplied tool named `recall` or `pin` is a configuration error
when `cognitive_primitives` is non-empty. `pool.add` raises `ValueError` at
construction time with a message naming the collision — we do not silently
shadow either direction.

## 3. Pool-level observability

### Trace event metadata

Every `TraceEvent` emitted during a pool run gets:

```python
event.metadata["source_agent"] = "<pool_agent_name>"
```

Implementation: `ChatSession.__init__` accepts an optional `_pool_agent_name`
parameter (set by `AgentPool._create_pool_session`); the session adds it to
`metadata` when emitting events. Sessions created outside a pool pass no value
and nothing changes in their trace output.

`TraceEvent` schema is unchanged. Existing `arcana trace` commands keep
working on v0.6.0/v0.7.0 traces.

### New CLI: `arcana trace pool-replay <run_id>`

```
arcana trace pool-replay <run_id>                  # summary of all agents
arcana trace pool-replay <run_id> --agent planner  # one agent's events
arcana trace pool-replay <run_id> --agent planner --turn 3
```

Builds on existing `TraceReader` infrastructure (filter events by
`metadata["source_agent"]`). Reuses the `PromptReplay` rendering path from
v0.6.0's `arcana trace replay`.

### Existing `arcana trace replay <run_id> --turn N`

Extended to accept an optional `--agent <name>` flag for pool runs. If omitted
on a pool trace, the CLI prints a usage hint listing the pool's agents.

### `arcana trace show --cognitive` (v0.7.0)

Already shows `COGNITIVE_PRIMITIVE` events; now also displays `source_agent`
in the per-event line:

```
 15. cognitive_primitive   2026-04-18T12:34  [planner]   recall    turn=3 → 2 messages (412 tokens)
 23. cognitive_primitive   2026-04-18T12:35  [executor]  pin       pin_id=p_abc label="result" (186 tokens)
```

## Known bugs & fix decisions (must ship in v0.8.0)

### Bug 1: `Runtime.collaborate()` `async with` breakage [P2]

Location: `src/arcana/runtime_core.py:1282-1286`

`collaborate()` is `async def`, returning a coroutine, but every usage example
uses `async with`. This raises `TypeError` at runtime.

**Fix** (decision: B from v080 memo — matches `chat()` pattern):
- Change `collaborate()` to a synchronous method returning `AgentPool`:
  ```python
  def collaborate(self, *, budget: Budget | None = None) -> AgentPool: ...
  ```
- `AgentPool` already has `__aenter__` / `__aexit__`
- The `BudgetTracker` construction inside `collaborate` stays synchronous

This aligns with `runtime.chat()` (sync factory, context-managed `ChatSession`).

### Bug 2: `Channel.send` broadcast shares mutable message [P2]

Location: `src/arcana/multi_agent/channel.py:27-31`

The same `ChannelMessage` instance is appended to `_history` and enqueued to
every recipient. Any mutation by one recipient (or `metadata` write) bleeds to
the others — classic shared-state bug.

**Fix** (decision: B from v080 memo — `ChannelMessage` immutable):
- Add `model_config = ConfigDict(frozen=True)` to `ChannelMessage`
- Add a regression test: two recipients see equal content; mutating through
  `model_copy(update=...)` on one copy does not affect the other

This is cheaper than per-recipient `model_copy(deep=True)` and matches the
Arcana contract-layer preference for immutable schemas.

## Constitutional check

| Check | Result | Notes |
|-------|--------|-------|
| Framework decides topology? | ❌ No | User code calls `pool.add(...)` and drives `.send()`; no graph, no DSL |
| Framework decides turn order? | ❌ No | No scheduler; user code picks who talks when |
| Framework dictates roles? | ❌ No | Role is whatever the user puts in `system="..."` |
| Framework imposes stop condition? | ❌ No | User code decides when to stop awaiting |
| Pool runs agents implicitly? | ❌ No | Nothing runs until user calls `await agent.send(...)` |
| Cognitive state shared across agents? | ❌ No | Per-session `PinState` / `CognitiveHandler` |
| Changes single-agent API? | ❌ No | Additive only |
| New prohibitions hit? | ❌ No | No premature structuring / theater / hoarding / retry |
| Breaks v0.7.0 consumers? | ❌ No | `TraceEvent` schema unchanged; `metadata["source_agent"]` is additive |

Principle 8 alignment: "the framework provides coordination infrastructure —
communication channels, budget allocation, turn scheduling — but never
dictates hierarchy or strategy." `AgentPool` delivers exactly the enumerated
infrastructure and nothing beyond.

## Four canonical patterns (user-code level, framework-neutral)

All four are pure user code on top of `pool`. None require new framework
surface. `docs/guide/multi-agent.md` covers each with a runnable example.

### 1. Planner / Executor (sequential)

```python
async with runtime.collaborate() as pool:
    planner  = pool.add("planner",  system="You plan tasks step by step.")
    executor = pool.add("executor", system="You execute steps.", tools=[shell])

    plan = await planner.send(f"Plan: {goal}")
    for step in parse_steps(plan.content):
        await executor.send(f"Execute step: {step}")
```

### 2. Critic loop (bounded iteration)

```python
async with runtime.collaborate(cognitive_primitives=["pin"]) as pool:
    worker = pool.add("worker", system="You write code.")
    critic = pool.add("critic", system="You review code.")

    draft = await worker.send(f"Write: {spec}")
    for _ in range(MAX_ITERS):
        review = await critic.send(f"Review:\n{draft.content}")
        if "LGTM" in review.content:
            break
        draft = await worker.send(f"Revise based on:\n{review.content}")
```

### 3. Specialization (different tools per agent)

```python
async with runtime.collaborate() as pool:
    researcher = pool.add("researcher", system="You research.",
                          tools=[web_search, fetch_url])
    writer     = pool.add("writer",     system="You write.",
                          tools=[notes_db])
    # …
```

### 4. Debate (channel-driven)

```python
async with runtime.collaborate() as pool:
    bull = pool.add("bull", system="Argue for X.")
    bear = pool.add("bear", system="Argue against X.")

    await bull.send("Open with your thesis.")
    pool.channel.send(ChannelMessage(sender="bull", recipient="bear",
                                     content=bull.last_response.content))
    # user code drives the rally; pool provides the bus
```

## Decisions (resolved from discussion)

| Decision | Resolution |
|----------|-----------|
| Primitive enablement granularity | Pool default + per-agent override via `pool.add(..., cognitive_primitives=...)` |
| Pool agents share cognitive state? | **No** — per-session isolation |
| `source_agent` on `TraceEvent` schema vs metadata | **metadata** — preserves JSONL schema stability |
| Pool budget vs per-agent pin budget | Shared pool budget for tokens/cost; per-agent `pin_budget_fraction` |
| `runtime.team()` status | Deprecated (already warns); removed in v1.0.0 |
| Tool name collision w/ cognitive primitive | `pool.add` raises `ValueError` at construction time |
| Bug 1 fix (`async with`) | Sync `collaborate()`; rely on `AgentPool.__aenter__` / `__aexit__` |
| Bug 2 fix (`Channel` shared message) | `ChannelMessage` immutable via `model_config = ConfigDict(frozen=True)` |
| Ship Tier 2 cognitive primitives (branch/anchor/hint)? | **No** — deferred to v0.9.0 |

## Open questions (escalate if blocking)

1. **Cross-agent `recall`** — should agent B be able to `recall` a turn from
   agent A's session? Current answer: **no** (per-session isolation). If
   users want cross-agent reference, they write `shared.set("A_turn_3", ...)`
   explicitly. Confirmed with Principle 8.

2. **Pool-level `pin_budget_fraction`** — single pool-wide value or per-agent
   setting? Tentative: per-agent, because pin budget is relative to each
   agent's own context window. `RuntimeConfig.pin_budget_fraction` stays the
   global default; per-agent override not exposed in v0.8.0 (wait for real
   demand).

3. **`AgentPool.add` after `__aenter__`** — is adding agents mid-run legal?
   Tentative: **yes**, it's just registry mutation. Test covers it.

4. **Channel history retention** — unbounded? For long-running pools this
   leaks memory. Tentative: add optional `Channel(history_limit=1000)`;
   default unlimited for backward compat. Confirm at implementation time.
