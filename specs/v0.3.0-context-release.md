# Arcana v0.3.0 "The Context Release" — Technical Specification

## Overview

Version 0.3.0 introduces **context transparency and adaptive management** to Arcana. Every LLM call produces a detailed `ContextReport` showing how the token budget was composed. The existing one-size-fits-all context compression is replaced with an **adaptive strategy system** that selects the appropriate compression level based on current utilization. Streaming responses gain **structured event types** to expose thinking, tool execution, and context composition in real-time.

These features make context management observable, configurable, and cost-effective while maintaining backward compatibility with existing code.

---

## Feature 1: Context Composition Report (ContextReport)

### 1.1 Overview

Every LLM call through `WorkingSetBuilder` produces a `ContextReport` that details how tokens were allocated across layers (identity, task, tools, history, memory) and how much compression saved. This report flows through the execution pipeline: `WorkingSetBuilder` → `ConversationAgent` → `RunResult` / `ChatResponse` / `TraceEvent` / `StreamEvent`.

### 1.2 Data Model

**File: `/Users/ty/self/agent/arcana/src/arcana/contracts/context.py`**

Add the following model at the end of the file:

```python
class ContextReport(BaseModel):
    """Report of how context was composed for a single LLM call.
    
    Produced by WorkingSetBuilder and attached to RunResult, ChatResponse,
    TraceEvent, and StreamEvent to provide full visibility into context
    window utilization.
    """
    
    # Turn metadata
    turn: int = 0
    strategy_used: str = "passthrough"  # "passthrough" | "tail_preserve" | "llm_summarize" | "aggressive_truncate"
    
    # Token allocation breakdown
    total_tokens: int = 0
    identity_tokens: int = 0      # System prompt
    task_tokens: int = 0           # Goal + context block from runtime.run(context=...)
    tools_tokens: int = 0          # Tool schemas (N/M loaded if lazy registry)
    history_tokens: int = 0        # Conversation history after compression
    memory_tokens: int = 0         # Injected memory facts
    
    # Compression metrics
    compression_applied: bool = False
    compression_savings: int = 0   # Tokens saved by compression (original - compressed)
    messages_compressed: int = 0   # Number of messages compressed
    
    # Utilization
    window_size: int = 128000
    utilization: float = 0.0       # total_tokens / window_size
    
    # Tool loading info (for lazy registry)
    tools_loaded: int = 0          # Number of tool schemas included
    tools_available: int = 0       # Total tools in registry (0 if not using lazy registry)
```

## Feature 2: Adaptive Compression Strategy

### 2.1 Overview

Replace the current fixed compression logic with a **strategy selection system** that adapts to context window utilization. Four strategies:

- **passthrough** (< 50% utilization): No compression, all messages intact
- **tail_preserve** (50-75%): Compress middle history, keep recent N turns verbatim (existing behavior)
- **llm_summarize** (75-90%): Use cheap LLM call for semantic summarization (existing LLM compression path)
- **aggressive_truncate** (> 90%): Keep only system + last 2 turns + goal

User can override with explicit thresholds or disable adaptive behavior.

### 2.2 Data Model

**File: `/Users/ty/self/agent/arcana/src/arcana/contracts/context.py`**

Add before `ContextReport`:

```python
class ContextStrategy(BaseModel):
    """Configuration for adaptive context compression strategy.
    
    Defines thresholds for when to apply different compression levels
    based on context window utilization ratio.
    """
    
    mode: str = "adaptive"  # "adaptive" | "off" | "always_compress"
    
    # Thresholds (as fraction of window_size)
    passthrough_threshold: float = 0.50   # Below this: no compression
    tail_preserve_threshold: float = 0.75  # 0.50-0.75: compress middle, keep tail
    llm_summarize_threshold: float = 0.90  # 0.75-0.90: use LLM summarization
    # Above 0.90: aggressive truncate
    
    # tail_preserve config
    tail_preserve_keep_recent: int = 6  # Number of recent messages to keep verbatim
    
    # aggressive_truncate config
    aggressive_keep_turns: int = 2  # Keep only last N user+assistant pairs
```

3. Add new `_aggressive_truncate()` method:
   
   ```python
   def _aggressive_truncate(
       self,
       messages: list[Message],
       *,
       turn: int,
       messages_in: int,
       effective_budget: int,
       effective_tool_tokens: int,
       memory_injected: bool,
   ) -> list[Message]:
       """Aggressive truncation: keep system + last N turns only.
       
       Used when context window is >90% full.
       """
       keep_turns = self._strategy.aggressive_keep_turns
       
       # System message (head)
       head = messages[:1] if messages else []
       
       # Last N user+assistant pairs (tail)
       # Count backward: look for alternating user/assistant pairs
       tail_messages = []
       turns_found = 0
       for msg in reversed(messages[1:]):
           tail_messages.insert(0, msg)
           if msg.role in (MessageRole.USER, "user"):
               turns_found += 1
               if turns_found >= keep_turns:
                   break
       
       result = head + tail_messages
       dropped_count = len(messages) - len(result)
       
       used_tokens = sum(estimate_tokens(_content_text(m.content)) for m in result)
       
       self.last_decision = ContextDecision(
           turn=turn,
           budget_total=effective_budget,
           budget_used=used_tokens,
           budget_tools=effective_tool_tokens,
           budget_reserve=self._budget.response_reserve,
           messages_in=messages_in,
           messages_out=len(result),
           compressed_count=dropped_count,
           memory_injected=memory_injected,
           history_compressed=True,
           compressed_messages=[f"[{dropped_count} messages dropped]"],
           explanation=f"Aggressive truncate: kept system + last {keep_turns} turns, dropped {dropped_count} messages",
       )
       
       return result
   ```

   ```python
           event_type=StreamEventType.CONTEXT_REPORT,
           run_id=run_id,
           step_id=str(state.current_step),
           metadata=context_report.model_dump(),
       )
   ```

5. Emit `TURN_END` at the end of each turn (before looping):
   
   ```python
   # At the end of the turn loop, after state update:
   yield StreamEvent(
       event_type=StreamEventType.TURN_END,
       run_id=run_id,
       step_id=str(state.current_step),
       turn_tokens=response.usage.total_tokens if response.usage else 0,
       turn_cost_usd=state.cost_usd,  # Cumulative cost
       budget_remaining_pct=(
           1.0 - (self.budget_tracker.total_cost / self.budget_tracker.max_cost_usd)
           if self.budget_tracker and self.budget_tracker.max_cost_usd > 0
           else None
       ),
   )
   ```

## Migration

### Backward Compatibility

**All changes are backward compatible:**

1. **ContextReport**: New optional field on `RunResult` and `ChatResponse`. Defaults to `None`. Existing code that doesn't check this field continues to work.

2. **ContextStrategy**: Default is `"adaptive"` with thresholds that match existing behavior at 50-75% utilization (tail_preserve). Below 50% is a new optimization (passthrough). Above 75% uses existing LLM compression. Above 90% is new aggressive mode but rarely triggered.

3. **StreamEvent types**: New event types are additive. Existing code that filters for `LLM_CHUNK` continues to work. `TEXT_DELTA` is emitted alongside `LLM_CHUNK` for backward compatibility.

4. **AgentState.last_context_report**: New optional field. Existing state serialization/deserialization handles missing fields gracefully (Pydantic default).

### Migration Path for Users

**No action required.** All features are opt-in or transparent:

- Context reports appear automatically in results; users can choose to inspect them or ignore them.
- Adaptive strategy is the default and improves performance under low utilization.
- Structured stream events are additive; existing event handlers continue to work.

### For users who want to opt out

```python
# Disable adaptive strategy (keep old behavior)
runtime = Runtime(
    providers={"deepseek": "sk-xxx"},
    context_strategy=ContextStrategy(
        mode="adaptive",
        passthrough_threshold=0.0,  # Always compress (never passthrough)
        tail_preserve_threshold=0.75,  # Existing threshold
        llm_summarize_threshold=1.0,  # Never use LLM summary
    ),
)
```

## Implementation Notes

### Token Estimation

Use the existing `estimate_tokens()` function in `context/builder.py`. It's approximate but consistent across the codebase.

### Strategy Selection Logic

The strategy selection happens at the **start** of each turn, based on current context window utilization. It does NOT look ahead to predict future utilization; this keeps the logic simple and deterministic.

### Lazy Tool Registry Integration

`ContextReport.tools_loaded` and `tools_available` are only populated when using `LazyToolRegistry`. When all tools are loaded eagerly, `tools_loaded == tools_available` (or both are 0 if no tools).

### LLM Compression Fallback

When `strategy == "llm_summarize"` but no gateway is configured (or LLM compression fails), the system falls back to `tail_preserve`. This is already implemented in the current code; no changes needed.

### Stream Event Timing

- `CONTEXT_REPORT`: Emitted **once per turn**, after context is built but before LLM call.
- `TOOL_START` / `TOOL_END`: Emitted **once per tool call**, in order of execution (parallel tools emit START in registration order, END in completion order).
- `TURN_END`: Emitted **once per turn**, after all tool calls and LLM response are complete.
