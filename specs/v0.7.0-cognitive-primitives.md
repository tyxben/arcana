# Arcana v0.7.0 "The Cognitive Primitives Release" — Technical Specification

**Base**: v0.6.0 (Explainability Release)
**Theme**: Runtime services that let the LLM operate on its own reasoning state
**Status**: 📝 PROPOSED (awaiting constitutional amendment review)
**Depends on**: Constitution Amendment 1 (`specs/constitution-amendment-1-cognitive-primitives.md`)

---

## Overview

Arcana v0.6.0 made the framework's context decisions transparent to the user. v0.7.0 makes them **operable by the LLM itself**.

Today the LLM is a passenger in its own context window — the working-set builder decides what to keep, compress, or drop; the LLM sees the result and reasons. This works for most tasks, but it leaves the LLM unable to:

- Read its own history if that history got compressed
- Protect its own conclusions from compression
- Explore a hypothesis without polluting main context
- Track its own assumptions across turns
- Signal what the next context should look like

v0.7.0 introduces five **cognitive primitives** as intercepted tools. Each primitive is a service the LLM can invoke; none are invoked by the framework. The MVP (this release) ships two.

**The Arcana positioning line becomes:**

> LangGraph / AutoGen / CrewAI give you an **orchestration OS** — ways to wire agents together.
> Arcana gives you a **cognition OS** — ways for the LLM to operate on its own mind.

| # | Name | Tool | Complexity | MVP? |
|---|------|------|-----------|------|
| 1 | Retrospective Probe | `recall(turn)` | Low | ✅ v0.7.0 |
| 2 | Pin | `pin(content, until_turn?)` | Medium | ✅ v0.7.0 |
| 3 | Hypothetical Frame | `branch_reasoning(label)` / `commit_branch` / `discard_branch` | High | v0.8.0 |

This spec details the two MVP primitives. Tier 2 primitives are described in skeleton form for roadmap visibility.

## Design invariants (apply to all primitives)

### 1. Intercepted-tool mechanism

Primitives are exposed as tools in the LLM's tool list. The LLM calls them by name. The runtime intercepts the call **before** `ToolGateway` dispatch — same pattern as `ask_user` (`runtime/conversation.py:_execute_tools`).

### 2. Opt-in via `RuntimeConfig`

```python
class RuntimeConfig(BaseModel):
    # ... existing fields ...
    cognitive_primitives: list[str] = []  # default: none enabled
```

Accepted values: `"recall"`, `"pin"`. Future: `"branch"`, `"anchor"`, `"hint"`.

Rationale:
- Unused primitives should not bloat tool schemas (anti Prohibition 3)
- Users choose which subset fits their task

### 3. Full trace emission

Every primitive invocation emits a `COGNITIVE_PRIMITIVE` trace event:

```python
class EventType(str, Enum):
    ...
    COGNITIVE_PRIMITIVE = "cognitive_primitive"
```

Metadata: `{"primitive": "recall" | "pin" | ..., "args": {...}, "result": {...}}`

Consumers (including `arcana trace show`) get full observability into when and how primitives are used.

### 4. Local to a session

Primitives operate on the **current session's** reasoning state. An `AgentPool` does not share cognitive state across agents. Each agent's pins are its own.

### 5. Never auto-invoked

The framework never calls a primitive on the LLM's behalf. Every invocation is an explicit LLM tool call with a tool_call_id and trace record.

## 1. Retrospective Probe — `recall`

### The problem

At turn N, the LLM may need the original text of a message from turn M (M < N). If the working set builder has compressed that message to L2 or L3, the LLM sees only a truncated stub. There is no way to retrieve the original content.

Example: turn 3 has a detailed plan. Turn 8 working set has plan at L3 ("[plan for project X]"). Turn 10 the LLM wants to check a specific step — it cannot.

### Contract

```python
# contracts/cognitive.py (new file)
class RecallRequest(BaseModel):
    turn: int                       # turn number to recall
    include: Literal["all", "assistant_only", "tool_calls"] = "all"

class RecallResult(BaseModel):
    turn: int
    found: bool
    messages: list[Message]          # full fidelity, from trace
    note: str | None = None          # e.g. "turn out of range"
```

### Tool schema (seen by LLM)

```python
ToolSpec(
    name="recall",
    description=(
        "Retrieve the full original content of a past turn, bypassing any "
        "working-set compression. Use when you need exact wording of an "
        "earlier message that may have been compressed or summarized."
    ),
    parameters={
        "type": "object",
        "properties": {
            "turn": {"type": "integer", "description": "Turn number (1-indexed)"},
            "include": {
                "type": "string",
                "enum": ["all", "assistant_only", "tool_calls"],
                "default": "all",
            },
        },
        "required": ["turn"],
    },
)
```

### Implementation

Interception in `ConversationAgent._execute_tools`:

```python
async def _execute_tool(self, call: ToolCall, ...) -> ToolResult:
    if call.name == "recall":
        return await self._handle_recall(call)
    if call.name == "pin":
        return await self._handle_pin(call)
    # ... existing ask_user / gateway dispatch
```

`_handle_recall` uses the existing `TraceReader` to fetch the turn's `LLM_REQUEST` or `LLM_RESPONSE` events and returns their full messages.

### Failure modes (Principle 5)

- `turn=999` when only 5 turns exist → `RecallResult(found=False, note="turn out of range: 999, max=5")`
- `turn=0` → validation error via structured tool result
- Trace unavailable → `note="trace not available for this run"`

All errors are returned as structured data the LLM can reason about, not raw exceptions.

### Relation to v0.6.0 `replay_prompt`

`recall` and `TraceReader.replay_prompt` solve related problems from opposite sides:

| | `recall` (v0.7.0) | `replay_prompt` (v0.6.0) |
|---|-----|-----|
| Who calls it | LLM (at runtime, via tool) | User (offline, via CLI or Python) |
| What it returns | Clean messages for LLM to read | Full `PromptReplay` with decision + snapshot |
| When it runs | Mid-conversation | After the fact |
| Trace dependency | Requires live trace writer | Requires finished trace file |

Both share the underlying `TraceReader` infrastructure. `_handle_recall` delegates to `TraceReader.events(run_id)` filtered by turn — no parallel trace-parsing code.

## 2. Pin — `pin`

### The problem

The LLM just derived three critical conclusions at turn 5. Without intervention, the working-set builder may compress them on turn 7 based on standard fidelity scoring. Token budget wins over semantic importance the LLM is aware of.

There is no way today for the LLM to say "these three lines are load-bearing — keep them at full fidelity."

### Contract

```python
class PinRequest(BaseModel):
    content: str                     # the exact content to pin
    label: str | None = None         # optional user-readable label
    until_turn: int | None = None    # auto-unpin after this turn; None = until session end

class PinResult(BaseModel):
    pin_id: str                      # UUID for later unpin
    label: str | None
    until_turn: int | None
```

### Tool schema

```python
ToolSpec(
    name="pin",
    description=(
        "Protect specific content from compression in future working sets. "
        "Use for critical conclusions you will need at full fidelity in "
        "subsequent reasoning. Returns a pin_id you can use to unpin later."
    ),
    parameters={
        "type": "object",
        "properties": {
            "content": {"type": "string", "description": "Exact content to pin at full fidelity"},
            "label": {"type": "string", "description": "Optional readable label"},
            "until_turn": {
                "type": "integer",
                "description": "Auto-expire after this turn; omit for session-long pin",
            },
        },
        "required": ["content"],
    },
)
```

Also exposes `unpin(pin_id)` in the same tool family.

### Working-set integration

**Principle 2's four-layer structure (Identity / Task / Working / External) is unchanged.** Pin is not a new layer — it is a **per-block flag** inside the Working layer.

Each pinned content becomes an independent `ContextBlock` in Working, marked `pinned=True`. The builder:

- Always includes pinned blocks at L0 fidelity
- Excludes pinned blocks from compression passes (`_compress_with_relevance` / `_aggressive_truncate`)
- Token-counts pinned blocks against the budget like any Working block
- Surfaces pinned blocks in `ContextDecision.decisions` with `outcome="kept"` and `reason="pinned"`

Pin does **not** match against existing messages. It inserts the pinned content as a new, independent block — even if similar text exists elsewhere in the conversation. This is the only reliable semantic (string matching on live messages would break under paraphrasing).

**Idempotency**: Pinning the same content twice is a no-op. The framework deduplicates by SHA-256 hash of the content string; the second call returns the existing `pin_id`.

### Budget interaction — hard cap, not soft warning

Pinned content gets a **hard budget cap**: `pinned_tokens <= total_budget * 0.5` (configurable via `RuntimeConfig.pin_budget_fraction: float = 0.5`).

Rationale: if pins could consume the entire budget, the Working layer would starve and the LLM would see only its own pinned content — a self-inflicted context hoarding failure.

Behavior when a new `pin` would exceed the cap:

```python
# Tool result returned to LLM (structured, Principle 5)
{
    "pinned": False,
    "reason": "pin_budget_exceeded",
    "current_pin_tokens": 3200,
    "requested_tokens": 800,
    "cap": 4000,
    "suggestion": "unpin some content (see active pins in next working set) or reduce new content size",
}
```

The LLM decides: unpin older content, reduce new pin size, or proceed without pinning. **The framework never auto-unpins.** Existing pins are never truncated — they remain at L0 or the pin call is rejected outright.

### Persistence

Pins live for the session's lifetime (or until `until_turn`). They are **not** persisted across runtimes — this is session-scoped cognitive state.

Trace records every pin/unpin, so offline replay can reconstruct pin state at any turn.

### Interaction with v0.6.0

- New `MessageDecision.outcome` value: *none needed* — pinned content uses existing `"kept"` with `reason="pinned"`
- `ContextDecision.decisions` now includes pinned entries (they still go through the decision pipeline, just always marked kept)
- `PromptSnapshot` captures the pin layer naturally since it dumps the full request

Backward compatible: old traces without pinned content just don't have pinned-reason entries.

## Tier 2 primitives (skeleton — not in v0.7.0)

### Hypothetical Frame — `branch_reasoning`

- `branch_reasoning(label)` → opens a sandboxed sub-session
- `commit_branch(label)` → merges branch's insights to main
- `discard_branch(label)` → discards branch entirely
- Implementation: sub-`ChatSession` with copy-on-write context; merge = selective message append

Complexity: branching context state, CoW semantics, merge conflict resolution. Non-trivial.

### Confidence Anchor — `anchor`

- `anchor(claim, depends_on=[turn_ranges_or_ids])` → records a provisional assumption
- Framework adds a passive watcher: if a future turn contradicts the claim (via LLM call or tool result), inject a diagnostic in the next working set
- Implementation: simple assumption store + turn-level diff checker

Needs: conflict detection heuristic (MVP: exact string disagreement; future: embedding similarity).

### Context Request — `request_context`

- `request_context(hint)` where hint is a structured preference ("include tool X docs next turn")
- Working-set builder consumes hints as input to strategy selection (not as rule)
- Purely advisory — builder may still override for budget reasons

## Constitutional check (this MVP)

| Check | Result |
|-------|--------|
| Helping or constraining the LLM? | Helping — gives recall access the LLM currently lacks |
| Framework-compelled usage? | No — opt-in config, LLM chooses whether to call |
| Changes existing strategy selection? | No — pinned flag is additive; compression logic unchanged for unpinned blocks; Principle 2's four-layer structure unchanged |
| New structured declarations? | No DSL — pin is a single content string, recall is a single turn int |
| Default on? | No — default empty list |
| PII / trace bloat risk? | Minimal — recall/pin events are small; content already in trace |
| Reversible? | Yes — LLM can unpin; session-scoped state |
| Breaks v0.6.0 consumers? | No — extends `ContextDecision.decisions` with one new reason; existing fields unchanged |

## CLI integration

Three touchpoints in existing `arcana trace` commands:

### `arcana trace show <run_id> --cognitive`

New filter flag. Shows only `COGNITIVE_PRIMITIVE` events, with human-readable formatting per primitive:

```
  15. cognitive_primitive   2026-04-18T12:34  recall    turn=3 → 2 messages (412 tokens)
  23. cognitive_primitive   2026-04-18T12:35  pin       pin_id=p_abc label="key conclusions" (186 tokens)
  41. cognitive_primitive   2026-04-18T12:37  unpin     pin_id=p_abc
```

### `arcana trace replay <run_id> --turn N`

Extended to surface **active pin state at turn N**. After the existing header + per-message table:

```
Active pins at turn N:
  p_abc  (label="key conclusions")  186 tokens  pinned at turn 5
  p_def  (no label)                  92 tokens  pinned at turn 8
```

This lets users debug "why is this content still in the prompt" or "why did this conclusion get lost" (answer: it was never pinned, got L3-compressed at turn M).

### `arcana trace show <run_id> --context`

Already shows context decisions. Extended so entries with `reason="pinned"` are flagged in output (e.g. `[PIN]` prefix), making it easy to see which messages were kept by pin vs by strategy logic.

## Decisions (resolved from review)

| Decision | Resolution |
|----------|-----------|
| Tool naming | `recall`, `pin`, `unpin` (no type suffixes — parameter names convey type) |
| Error semantics | Structured tool result (Principle 5) |
| Pin `label` | Optional |
| Multi-primitive in one turn | Allowed (parallel tool execution already supported) |
| Unpin mechanism | By `pin_id` (returned from `pin` call) |
| Pin storage | Independent `ContextBlock` with `pinned=True` flag inside Working layer — **not** a new layer |
| Pin idempotency | SHA-256 content hash; duplicate pin returns existing `pin_id` |
| Pin budget cap | Hard cap at 50% of total budget (`RuntimeConfig.pin_budget_fraction`) |
| Over-cap behavior | Reject new pin via structured tool result; never auto-unpin or truncate existing pins |
